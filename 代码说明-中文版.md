# PaiAgent 代码说明（中文版）

这份文档面向“想读懂这个项目怎么工作的学习者”，重点不是把所有文件逐个翻译一遍，而是帮你建立一条清晰主线：

1. 前端如何让用户搭建工作流
2. 数据如何保存到后端
3. 后端如何把节点图解析成可执行流程
4. 一个工作流是怎样真正跑起来的

---

## 1. 项目是什么

PaiAgent 是一个“可视化 AI 工作流平台”。

你可以把它理解成一个简化版的工作流编排系统：

- 前端提供一个拖拽画布
- 用户在画布里放入节点，例如 `INPUT`、`LLM`、`TOOL`、`OUTPUT`
- 用户把节点连起来，形成有向图
- 后端把这张图解析成 DAG（有向无环图）
- 然后按拓扑顺序执行每一层节点

当前仓库分成两个模块：

- `paiagent-frontend/`：React + TypeScript + Vite 前端
- `paiagent-backend/`：Spring Boot + JPA + Flyway 后端

## 3. 前端怎么组织

### 3.1 前端入口很薄

### 3.2 页面骨架：一个典型的“工作台”布局

[paiagent-frontend/src/components/layout/AppLayout.tsx](/D:/AI/agentTest/paiagent-own/paiagent-frontend/src/components/layout/AppLayout.tsx) 负责拼装整个界面：

- 右侧：`NodeConfigPanel`
- 底部抽屉：`DebugDrawer`

同时它包了一层 `ReactFlowProvider`，说明画布核心依赖 `reactflow`。

你可以把这一层理解成“编辑器外壳”。

### 3.3 画布核心：React Flow + Zustand

[paiagent-frontend/src/components/canvas/FlowCanvas.tsx](/D:/AI/agentTest/paiagent-own/paiagent-frontend/src/components/canvas/FlowCanvas.tsx) 是画布最关键的组件。

它做了几件事：

- 从 `useFlowStore` 读取 `nodes` 和 `edges`
- 把这些数据交给 `ReactFlow`
- 处理拖拽放置新节点
- 处理节点连线
- 设置默认边样式、背景、缩略图、控制器

这里有一个很重要的学习点：

前端没有自己造一套画布系统，而是直接把“节点图编辑”交给 React Flow，把“状态更新”交给 Zustand。这是很典型的工程化做法。

### 3.4 节点状态保存在 `useFlowStore`

[paiagent-frontend/src/stores/useFlowStore.ts](/D:/AI/agentTest/paiagent-own/paiagent-frontend/src/stores/useFlowStore.ts) 是前端最重要的状态文件之一。

它维护：

- `nodes`
- `edges`
- 节点变化处理函数 `onNodesChange`
- 连线变化处理函数 `onEdgesChange`
- 连接事件 `onConnect`
- 新增节点 `addNode`
- 更新节点数据 `updateNodeData`
- 删除节点 `removeNode`

几个关键点：

- 初始时默认就有一个输入节点和一个输出节点
- 新节点 ID 用 `nanoid` 生成
- 节点展示类型会根据业务类型映射到 React Flow 节点类型
- 删除节点时会顺带删除相关连线

这说明：前端画布的“真数据源”不是组件内部状态，而是 Zustand Store。

### 3.5 顶部工具栏负责“新建 / 保存 / 加载 / 调试”

[paiagent-frontend/src/components/layout/TopNavBar.tsx](/D:/AI/agentTest/paiagent-own/paiagent-frontend/src/components/layout/TopNavBar.tsx) 是你理解前后端交互最重要的前端文件。

它完成这些动作：

- `handleNew()`：清空当前画布
- `handleSave()`：把当前 `nodes + edges` 转换成后端需要的 DTO，然后调用保存接口
- `handleLoad()`：先获取工作流列表
- `handleLoadWorkflow()`：拿到指定工作流后，把后端返回的数据重新组装为 React Flow 所需结构

这一层本质上做的是：

`前端画布数据结构 <-> 后端持久化 DTO`

这是学习这类项目时必须看懂的一层，因为很多“为什么前后端字段长这样”的答案都在这里。

### 3.6 节点配置面板负责编辑节点参数

[paiagent-frontend/src/components/config-panel/NodeConfigPanel.tsx](/D:/AI/agentTest/paiagent-own/paiagent-frontend/src/components/config-panel/NodeConfigPanel.tsx) 用于编辑当前选中节点的配置。

它的本质不是“保存配置”，而是“修改当前节点的 `data.config`”。真正持久化发生在点击保存之后。

[paiagent-backend/src/main/java/com/paiagent/service/WorkflowService.java](/D:/AI/agentTest/paiagent-own/paiagent-backend/src/main/java/com/paiagent/service/WorkflowService.java) 负责：

- 查询工作流
- 创建工作流
- 更新工作流
- 删除工作流
- Entity 和 Response DTO 之间的转换

这里值得重点理解两件事：

1. `applyRequest()`  
它会把前端传来的节点和边，转成 `WorkflowNode` 与 `WorkflowEdge` 实体。

2. `toJson()` / `parseJson()`  
节点配置、画布状态这类结构化但不固定的内容，被保存成 JSON 字符串。

这说明项目设计上把“固定结构”和“灵活配置”分开了：

- 固定字段用数据库列
- 灵活配置用 JSON

这是一种很常见的低成本设计。

### 5.4 执行服务

[paiagent-backend/src/main/java/com/paiagent/service/ExecutionService.java](/D:/AI/agentTest/paiagent-own/paiagent-backend/src/main/java/com/paiagent/service/ExecutionService.java) 是“执行流程”的总调度入口。

它主要负责：

- 加载工作流及其节点、边
- 创建 `ExecutionLog`
- 调用 `WorkflowEngine`
- 记录成功/失败状态
- 在流式执行模式下通过 `SseEmitter` 推送节点状态

你可以把它理解成：

`执行事务层 + 日志记录层 + SSE 事件转发层`

## 6. 后端执行引擎怎么工作

这是整个项目最核心的部分。

### 6.1 `WorkflowEngine` 是总入口

[paiagent-backend/src/main/java/com/paiagent/engine/WorkflowEngine.java](/D:/AI/agentTest/paiagent-own/paiagent-backend/src/main/java/com/paiagent/engine/WorkflowEngine.java) 的执行过程很清晰：

1. 把数据库里的节点和边解析为 DAG
2. 对 DAG 做拓扑排序
3. 创建执行上下文 `ExecutionContext`
4. 按阶段（stage）执行节点
5. 找到输出节点，返回结果

注意这里的“按阶段执行”很重要，表示它不是完全串行，而是：

- 有依赖关系的节点必须按顺序
- 同一层没有依赖关系的节点可以并行

### 6.2 `ExecutionContext` 是运行时数据总线

[paiagent-backend/src/main/java/com/paiagent/engine/ExecutionContext.java](/D:/AI/agentTest/paiagent-own/paiagent-backend/src/main/java/com/paiagent/engine/ExecutionContext.java) 虽然这次没有单独展开看，但从执行器调用方式已经能看出它的职责：

- 保存用户原始输入
- 保存每个节点执行后的输出
- 让后续节点通过节点 ID 读取前置结果

你可以把它理解成“运行时内存中的共享结果表”。

### 6.3 四类节点分别由不同执行器处理

后端使用了策略模式思想，不同节点类型由不同 `NodeExecutor` 实现处理：

- [InputNodeExecutor.java](/D:/AI/agentTest/paiagent-own/paiagent-backend/src/main/java/com/paiagent/engine/executor/InputNodeExecutor.java)
- [LLMNodeExecutor.java](/D:/AI/agentTest/paiagent-own/paiagent-backend/src/main/java/com/paiagent/engine/executor/LLMNodeExecutor.java)
- [ToolNodeExecutor.java](/D:/AI/agentTest/paiagent-own/paiagent-backend/src/main/java/com/paiagent/engine/executor/ToolNodeExecutor.java)
- [OutputNodeExecutor.java](/D:/AI/agentTest/paiagent-own/paiagent-backend/src/main/java/com/paiagent/engine/executor/OutputNodeExecutor.java)

[paiagent-backend/src/main/java/com/paiagent/engine/NodeExecutor.java](/D:/AI/agentTest/paiagent-own/paiagent-backend/src/main/java/com/paiagent/engine/NodeExecutor.java) 是它们共同实现的接口。

#### INPUT 节点

`InputNodeExecutor` 的逻辑最简单：

- 读取用户输入
- 存到 `ExecutionContext`
- 返回输入文本

#### LLM 节点

`LLMNodeExecutor` 会：

- 找到当前节点的所有前驱节点
- 把前驱输出拼接成输入文本
- 从节点配置中读取 `baseUrl/apiKey/model/systemPrompt/temperature/maxTokens`
- 根据 `subtype` 找到对应的 LLM 适配器
- 调用模型接口
- 把结果写回 `ExecutionContext`

所以 LLM 节点的真正工作流程是：

`读取上游结果 -> 组装请求 -> 调模型 -> 写回上下文`

#### TOOL 节点

`ToolNodeExecutor` 跟 LLM 节点很像，只不过它不调用模型适配器，而是调用工具插件：

`读取上游结果 -> 找工具插件 -> 执行 -> 写回上下文`

#### OUTPUT 节点

`OutputNodeExecutor` 会把所有前驱节点结果拼接起来，作为最终输出。

### 6.4 并行执行器

[paiagent-backend/src/main/java/com/paiagent/engine/ParallelStageExecutor.java](/D:/AI/agentTest/paiagent-own/paiagent-backend/src/main/java/com/paiagent/engine/ParallelStageExecutor.java) 负责执行同一阶段的节点。

关键点：

- 单节点阶段直接执行
- 多节点阶段用 `CompletableFuture` 并发执行
- 底层线程池使用 Java 21 虚拟线程
- 流式执行时会通过监听器回调每个节点的开始、成功、失败事件

这部分体现了项目一个比较现代的点：利用 Java 21 虚拟线程提升并发执行可读性和成本效率。

## 11. 一次“保存工作流”的完整流程

当你在前端点击保存时，大致会发生这些事：

1. `TopNavBar.handleSave()` 从 `useFlowStore` 取出当前节点和边
2. 把 React Flow 的节点结构转换成 `WorkflowDTO`
3. 调用前端 `workflowApi.create()` 或 `workflowApi.update()`
4. 请求进入后端 `WorkflowController`
5. `WorkflowService.applyRequest()` 把 DTO 转成 `Workflow / WorkflowNode / WorkflowEdge`
6. JPA 保存到 MySQL
7. 后端返回 `WorkflowResponse`
8. 前端把返回的工作流 ID 存起来，后续就能继续更新

你可以把这条链路当成这个项目最基础的一条主线。

---

## 12. 一次“执行工作流”的完整流程

当用户执行工作流时，核心流程是：

1. 前端调用 `/api/v1/workflows/{workflowId}/execute`
2. `ExecutionController` 接到请求
3. `ExecutionService` 加载工作流、创建执行日志
4. `WorkflowEngine` 将节点边解析成 DAG
5. `TopologicalSorter` 生成分阶段执行计划
6. `ParallelStageExecutor` 按 stage 执行节点
7. 各类 `NodeExecutor` 分别处理输入、模型、工具、输出
8. 结果写入 `ExecutionContext`
9. 最终从 `OUTPUT` 节点取出结果
10. 结果写回 `execution_logs` 并返回前端

如果你想真正读懂这个项目，这条链路最好自己顺着代码再走一遍。

## 14. 学这个项目时最值得关注的设计点

### 14.1 前后端的数据结构映射

React Flow 的节点边结构和数据库结构不是一回事，所以项目专门做了一层 DTO 映射。这个思路在很多低代码、流程编排项目里都很常见。

### 14.2 执行器模式

不同节点类型对应不同执行器，而不是用一大堆 `if-else` 写死在一个类里。这样扩展新节点时会更干净。

### 14.3 DAG + 拓扑排序

这是工作流系统的经典实现方式。只要你理解了这部分，就能举一反三看懂很多任务编排系统。

### 14.4 JSON 配置存储

节点配置变化大，所以用 JSON 存数据库，比频繁改表结构更灵活。

### 14.5 插件化扩展

LLM 和 Tool 都用了注册器模式，说明作者已经在为后续扩展预留结构。
